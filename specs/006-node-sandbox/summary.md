# Summary: Spec 006 Remote Node Runner and Per-Agent Sandboxing

Spec 006 changes `python-claw` by separating privileged command execution from the main gateway and worker runtime. Before this slice, the project had no real remote-execution boundary. After this slice, privileged execution is represented as a typed, approval-gated capability called `remote_exec`, and the actual host-side execution happens through a separate node-runner service contract instead of being constructed inline inside graph nodes. This is an important architecture step because it keeps orchestration logic and dangerous execution logic in different places.

The biggest idea in this spec is that remote execution must be fail-closed. The gateway or worker cannot just send a free-form shell string to a subprocess. Instead, execution has to come from an approved `node_command_template`, exact invocation parameters, deterministic argv derivation, and one signed `NodeExecRequest`. That request is then verified again by the node runner before anything runs. If the signature is wrong, the request is stale, the executable is not allowlisted, the sandbox data does not match, or the payload was tampered with, the request is rejected before execution starts.

Another important change is replay-safe execution identity. Spec 005 already introduced durable run identity for queued work, and Spec 006 builds on that by defining one logical remote-execution attempt with `(execution_run_id, tool_call_id, execution_attempt_number)`. From those values, the runtime derives a stable `request_id`. That means duplicate delivery of the same execution attempt must reuse the same audit row and must never start a second process. This is a safety feature, not just a convenience feature, because it prevents retries or network duplication from accidentally executing the same privileged action twice.

This spec also adds the first explicit sandbox model for execution. Each agent can now have an `agent_sandbox_profiles` row that defines default sandbox behavior, whether `off` mode is allowed, which shared profile key to use, and the maximum timeout. The runtime then resolves a sandbox mode of `off`, `shared`, or `agent`, plus a deterministic sandbox key and workspace root for the specific request. In the current workspace, the metadata and resolution logic are implemented, but the container backend is still only scaffolded. So the system now has the contracts and auditability for sandboxing even though full container isolation is not yet the finished production path.

To support operations and debugging, Spec 006 introduces durable execution auditing through `node_execution_audits`. This table records the stable `request_id`, execution identity, sandbox metadata, workspace data, terminal status, previews of stdout and stderr, and denial reasons when execution fails closed. That gives developers and operators a trustworthy place to inspect what happened during a remote execution attempt without relying only on logs or in-memory state.

Finally, this spec preserves the existing governance and tool architecture instead of bypassing it. Remote execution is not a special escape hatch. It still uses typed actions, exact-match approvals from Spec 003, worker ownership from Spec 005, and bounded continuity-facing artifacts from the Spec 002 and Spec 004 layers. In other words, Spec 006 does not add "run arbitrary shell"; it adds a governed, signed, replay-safe execution path with audit records and sandbox metadata that fits the same architectural rules as the earlier specs.
