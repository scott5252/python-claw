# Tasks 006: Remote Node Runner and Per-Agent Sandboxing

1. Define the canonical remote-execution contracts in `src/execution/contracts.py` and `src/security/signing.py`, including `node_command_template` workspace-binding fields, `NodeExecRequest` fields for `tool_call_id`, `execution_attempt_number`, `workspace_root`, and `workspace_mount_mode`, stable `request_id` derivation from `(execution_run_id, tool_call_id, execution_attempt_number)`, lifecycle state mappings, and explicit rejection rules for malformed, stale, or tampered requests.
2. Write high-risk contract tests for canonicalization and verification invariants before execution handlers exist, covering stable request hashing, stable same-attempt `request_id` reuse, changed `request_id` after durable attempt advancement, unknown `key_id` rejection, freshness-window enforcement, replay protection for same-`request_id` different-payload submissions, and mismatch rejection between approved template data, canonical params, derived `argv`, and resolved workspace data.
3. Add additive persistence for `node_execution_audits` and `agent_sandbox_profiles`, including enums, unique and lookup indexes, and repository contracts for `insert-or-get` by `request_id`, transition updates, read-only status lookup, persisted `tool_call_id`, `execution_attempt_number`, `workspace_root`, and `workspace_mount_mode`.
4. Write repository tests for durable idempotency and audit safety, covering duplicate delivery returning the existing row in any state, terminal and non-terminal transition persistence, fail-closed behavior when audit writes or uniqueness guarantees cannot be satisfied, and canonical remote-exec tool outcome persistence keyed by `request_id` for continuity replay.
5. Implement gateway-side approval and policy enforcement in `src/capabilities/repository.py`, `src/policies/service.py`, and `src/tools/registry.py` so remote execution tools remain deny-by-default and are only exposed when remote execution is enabled, policy allows the capability, and an exact matching approval plus immutable resource version is available.
6. Add gateway/runtime tests for deny-by-default and approval enforcement, proving the remote execution tool is omitted when configuration is off, approvals are absent or revoked, policy data is stale, or execution-time approval refresh for queued work no longer matches the approved resource and canonical parameters.
7. Implement the gateway-owned remote execution runtime in `src/execution/runtime.py`, `src/tools/remote_exec.py`, `src/jobs/repository.py`, and `src/sessions/concurrency.py` so it refreshes execution-time policy and approval state, loads the immutable approved `node_command_template`, deterministically derives canonical params plus final `argv`, resolves sandbox mode and key, resolves canonical workspace identity and mount mode, signs one `NodeExecRequest`, and ties terminal tool-outcome ownership to the active Spec 005 lease.
8. Add runtime and worker-ownership tests proving transport retries reuse the same `request_id`, later durable attempts produce a different `request_id`, a worker that loses the active lease cannot persist the terminal tool outcome, and a resumed worker can reconcile by `request_id` without producing a second terminal outcome.
9. Implement node-runner policy verification in `apps/node_runner/policy.py`, enforcing signature verification, freshness checks, duplicate and replay handling, exact approved-template consistency checks, executable allowlists, shell-wrapper denial, timeout ceilings, sandbox-mode validation, workspace-resolution consistency checks, and fail-closed rejection before any sandbox allocation or process start.
10. Add node-runner policy tests for host-safety denial behavior, including disallowed executables, shell interpreter wrappers, disallowed `off` mode, env or working-directory overrides outside the approved template, timeout escalation beyond approved ceilings, shared-mode workspace write attempts, and duplicate delivery that must reuse persisted state instead of starting a second execution.
11. Implement sandbox profile resolution and container-backed isolation in `src/sandbox/service.py` and `src/sandbox/backends/container.py`, covering deterministic `off`, `shared`, and `agent` selection keys, deterministic workspace resolution for `agent`, `session`, and `fixed` bindings, explicit backend descriptor lookup, read-only root filesystem defaults, bounded writable mounts, read-only workspace enforcement for `shared`, env filtering, disabled outbound network by default, explicit runtime user identity, and no silent downgrade on allocation failure.
12. Add sandbox resolution and isolation tests covering deterministic mode and key selection, deterministic workspace resolution, rejection when deployment config or agent profile disallows `off`, refusal to downgrade failed `agent` or `shared` allocation, read-only enforcement for `shared` workspace mounts, and fail-closed behavior when sandbox descriptors or backend dependencies are unavailable.
13. Implement node-runner execution and audit transitions in `apps/node_runner/executor.py`, `src/execution/audit.py`, `src/sessions/repository.py`, and `apps/node_runner/api/internal.py`, ensuring the audit row is inserted or reused before acknowledgment, bounded stdout and stderr previews are captured, lifecycle states move through `received`, `running`, and terminal outcomes correctly, `POST` plus `GET` surfaces return the persisted status for polling or duplicate callers, and the canonical Spec 002 remote-exec tool outcome artifact stores `request_id`, terminal status, failure category, and bounded diagnostics for continuity replay.

15. Write API and integration tests for node health and lifecycle behavior, covering startup failure when critical config is missing, successful signed execution through gateway to node runner to audit completion, unsigned or stale request rejection before execution, replay rejection for tampered duplicates, approval revocation before queued execution, timeout handling, sandbox-backend failure without downgrade, node unavailability surfacing as a worker-visible failure without local fallback, resumed-worker reconciliation after worker loss, and duplicate delivery producing one logical execution attempt and one recorded tool outcome.
