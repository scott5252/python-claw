# Summary: Spec 005 Async Queueing

Spec 005 changes `python-claw` from a request-bound execution model to a durable accept-and-queue model for user-visible turns. Before this slice, the gateway accepted an inbound message and ran the assistant graph inline before returning. After this slice, the gateway now persists the inbound transcript row, creates or reuses a durable `execution_runs` record, finalizes dedupe state, and returns `202 Accepted` with a `run_id`. That means the request path only promises durable acceptance and queue submission, while the actual graph execution happens later in a worker-owned path.

The main accomplishment of this spec is separating gateway responsibilities from worker responsibilities without breaking the earlier guarantees from Specs 001 through 004. The gateway still owns routing validation, session resolution, transcript-first persistence, and duplicate suppression. The worker now owns claiming runs, executing the graph, classifying failures, retrying transient problems, and recording terminal outcomes. This keeps the architecture gateway-first and transcript-first while making long-running turns safe to process outside the HTTP request lifecycle.

Spec 005 also introduces the durable queueing and concurrency primitives needed to make async execution safe. `execution_runs` is now the canonical queue for user-visible work, and session-lane leases ensure that only one active run can execute at a time for a given session. On top of that, the implementation adds a global concurrency cap so the system can bound how many graph runs are active at once across all sessions. FIFO ordering within a session lane is part of the contract, so later turns in the same conversation cannot overtake earlier accepted work.

Another major part of the slice is making recovery and replay deterministic. Duplicate inbound deliveries must resolve to the same logical run, not create a second one. Worker crashes and expired leases must leave runs recoverable rather than stranded. Scheduler-triggered work follows the same gateway-owned path by persisting fire records, creating canonical scheduler-authored trigger messages, and enqueuing work through the same queue contracts instead of bypassing the runtime boundary. This keeps replay, auditing, and provenance consistent whether work came from a user message or a scheduled job.

Finally, this spec adds the operational visibility needed to work with async execution. The system now exposes read-only run diagnostics through `GET /runs/{run_id}` and `GET /sessions/{session_id}/runs`, so developers and operators can inspect queue state, attempts, timestamps, and failures without mutating anything. Just as important, Spec 005 deliberately preserves Spec 004 `outbox_jobs` as separate post-commit derived-state work. In other words, the new queue handles user-visible turn execution, while summaries, retrieval indexing, and repair jobs remain a distinct background concern.
