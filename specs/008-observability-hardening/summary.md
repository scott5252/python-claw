# Summary: Spec 008 Observability, Diagnostics, and Operational Hardening

Spec 008 makes `python-claw` much easier to operate as a real service by adding the observability and safety features that were still missing after Specs 001 through 007. Before this slice, the system could accept work, queue runs, normalize attachments, execute tools, and record durable state, but operators still had to piece together what happened from a small health endpoint, a few narrow read routes, and raw database inspection. After this slice, the platform has a clearer operational model: each accepted run gets a stable correlation identity, health and readiness are split into separate responsibilities, and authenticated diagnostics routes expose the most important state in a bounded, structured way.

One of the biggest changes is the introduction of a stable end-to-end `trace_id` for execution runs. In this spec, `execution_runs.trace_id` becomes the canonical identifier used to follow one accepted run across the gateway, worker, context assembly, node execution, attachment normalization, and outbound delivery. That matters because earlier slices created durable records for all of those stages, but there was not yet one shared correlation key that tied them together consistently. The gateway now mints or repairs the trace identifier when a run is created or reused, and downstream work reuses that same value instead of inventing component-local identifiers. For junior developers, the easiest way to think about this is: if you want to understand one run from start to finish, `trace_id` is now the thread you follow.

Spec 008 also adds a shared observability layer instead of letting each module log or classify failures in its own ad hoc way. The code now includes helpers for structured event logging, redaction, failure categorization, lightweight tracing and metrics hooks, and health evaluation. This means logs are more machine-friendly and safer to expose to operators because required fields such as `event_name`, `trace_id`, `component`, `status`, and other context are emitted in a consistent format, while sensitive material such as bearer tokens and secrets is redacted. Just as importantly, failure categories like `delivery_failed`, `dependency_unavailable`, or `unexpected_internal` are now persisted and reused across diagnostics so the system reports problems in a more predictable way.

Another major improvement is the new health and diagnostics API surface. Instead of one overloaded `GET /health` route, the system now distinguishes `GET /health/live` from `GET /health/ready`. Liveness answers the simple question "is this process up," while readiness answers the more operational question "is this service safe to receive new traffic right now." On top of that, the gateway now exposes authenticated `/diagnostics/*` routes for runs, session continuity, outbox jobs, node executions, deliveries, and attachments. These routes all use a shared bounded response shape with cursor-based pagination, which is important because it gives operators useful visibility without turning diagnostics into an unbounded dump of production state.

This spec also extends the durable data model so observability is tied to persisted truth instead of being only a logging feature. `execution_runs`, `outbox_jobs`, `outbound_deliveries`, and `outbound_delivery_attempts` now carry more correlation and failure metadata, and diagnostics read from those tables directly. That design choice is important for the architecture of the project: it keeps the platform gateway-first and transcript-first, and it avoids creating a second shadow system of operational state that could drift from the canonical records. In practical terms, if a delivery failed, a run degraded, or an outbox job got stuck, operators can now inspect that using durable records and diagnostics endpoints instead of guessing from scattered logs.

Finally, Spec 008 keeps the scope intentionally practical. It does not add flashy new end-user behavior, provider failover, or full production telemetry infrastructure. Instead, it adds the minimum hardening needed to safely run the capabilities already delivered in earlier specs: deny-by-default diagnostics auth, bounded diagnostics payloads, stale-work visibility, local development support with no-op metrics or tracing, and explicit degraded-state reporting. For junior developers, this is a good example of how a mature backend evolves: once the core features exist, the next important step is making the system understandable, debuggable, and safe to operate under failure, not just adding more product features.
