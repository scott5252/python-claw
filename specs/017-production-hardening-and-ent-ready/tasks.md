# Tasks 017: Production Hardening and Enterprise Readiness

## Implementation Readiness Review

- The spec is implementable on the current codebase because Specs 001 through 016 already established the durable ownership seams this slice needs to harden rather than replace:
  - Spec 001 made the gateway the sole inbound path and `sessions` plus append-only `messages` the canonical conversation record
  - Specs 002, 009, and 010 already separate orchestration, provider-backed runtime behavior, and typed-tool execution from transport
  - Spec 003 made approval and activation backend-authoritative, so production hardening can extend callback and admin protections without inventing a second approval authority
  - Spec 005 made `execution_runs` the durable queue owner with one-row retry semantics
  - Specs 007, 012, and 013 already split outbound intents, deliveries, delivery attempts, and streaming behavior in a way that supports durable delivery-owned retry
  - Spec 006 established the signed node-runner and sandbox boundary that this slice must strengthen with transport auth, mode handling, and bounded output capture
  - Spec 008 already introduced logging, health, diagnostics, metrics, tracing, stale thresholds, and redaction facades, even though several of them are still placeholder implementations
  - Specs 014 through 016 added durable agent bindings, delegation, approval prompts, and collaboration state that recovery and auth changes must preserve exactly
- The current runtime seams line up well with Spec 017:
  - `apps/gateway/deps.py` already centralizes auth helpers, service construction, provider runtime wiring, and node-runner composition
  - `apps/gateway/api/admin.py` and `apps/gateway/api/health.py` already expose the sensitive read and write surfaces that need one shared dual-caller auth contract
  - `src/config/settings.py` already validates real-vs-fake channel accounts, provider runtime settings, delegation limits, and stale thresholds, so it is the natural home for the new production-hardening settings
  - `src/jobs/repository.py` and `src/jobs/service.py` already own queue claim, retry, blocked-run release, and worker execution, which makes them the right seams for quota-safe retry, recovery-safe requeue, and stale-run repair
  - `src/channels/dispatch.py` already owns durable delivery creation and append-only attempts, which is the right place to encode durable delivery retry ownership and redrive behavior
  - `src/providers/models.py` already classifies provider failures and performs bounded retries, but still does so with adapter-local defaults rather than one explicit production retry contract
  - `src/observability/metrics.py`, `src/observability/tracing.py`, `src/observability/health.py`, `src/observability/diagnostics.py`, `src/observability/logging.py`, and `src/observability/redaction.py` already provide the façade layer this slice should harden
  - `apps/node_runner/api/internal.py`, `src/execution/runtime.py`, `src/security/signing.py`, `src/sandbox/service.py`, and `apps/node_runner/policy.py` already form the node-runner trust boundary that now needs explicit mode-aware transport auth and rotation support
- The highest-risk implementation failures to prevent are:
  - leaving admin, transcript, governance, diagnostics, or readiness reads open because only some routes use the current token helper
  - allowing internal-service credentials to satisfy operator-authored mutations or to persist placeholder principals like `internal-service` into human audit fields
  - enforcing rate limits after dedupe, transcript append, run creation, or provider dispatch and thereby producing durable side effects for rejected traffic
  - retrying provider or delivery failures by creating new logical runs or new logical delivery rows instead of reusing the existing durable identities
  - implementing stale-work recovery from opportunistic read paths or user-input replay rather than from one explicit bounded recovery owner
  - surfacing raw bearer tokens, signing secrets, provider keys, channel secrets, or secret-bearing headers in logs, diagnostics, health output, traces, or settings serialization
  - hardening node-runner HTTP transport in a way that breaks the existing in-process execution path used by tests and local development
  - emitting unbounded-cardinality metrics or quota scopes keyed by arbitrary user ids or payload data
  - introducing diagnostics or recovery scans that require unbounded table reads to explain current state
  - shipping smoke coverage that still depends on live third-party systems and therefore cannot protect CI or local production-readiness validation
- The tasks below are ordered to prove the hardest production invariants first, then add the additive schema and shared contracts, then wire quotas, retries, recovery, telemetry, and node-runner hardening through the existing architecture, and finally land docs and deterministic smoke validation.

- Coverage against the spec is complete:
  - shared dual-caller auth across admin, diagnostics, readiness, and node-runner surfaces
  - settings-backed credential validation, masking, and rotation overlap support
  - PostgreSQL-backed quota counters and bounded rate-limit enforcement
  - provider retry and delivery retry ownership on existing durable identities
  - one explicit recovery service and typed stale-state matrix
  - exporter-backed metrics and tracing behind the existing façade layer
  - bounded diagnostics, alertable operational signals, and retention-aware operator views
  - sandbox, remote execution, stdout or stderr, and media-retention hardening
  - deterministic smoke coverage plus disaster-recovery documentation
- Coverage against the current codebase is concrete:
  - tasks anchor auth and principal work in `apps/gateway/deps.py`, `apps/gateway/api/admin.py`, `apps/gateway/api/health.py`, and `apps/node_runner/api/internal.py`
  - tasks anchor quota, retry, and recovery work in `src/jobs/`, `src/channels/dispatch.py`, `src/providers/models.py`, and the existing worker loop
  - tasks preserve the existing façade and diagnostics seams in `src/observability/`
  - tasks harden the current node-runner and sandbox boundary instead of replacing it
- The task list should support successful implementation of Spec 017 because it forces the team to prove the hardest production invariants first:
  - auth fail-closed behavior and operator vs internal-service separation
  - pre-mutation quota rejection with no durable side effects
  - one-run and one-delivery retry ownership under transient failure
  - idempotent stale-work repair without replaying user input
  - strict secret non-exposure across logs, diagnostics, health, and settings posture
  - deterministic production-readiness validation through local smoke and recovery scenarios
