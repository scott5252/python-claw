# Tasks 009: Provider-Backed LLM Runtime

1. Confirm the existing graph, policy, repository, and runtime tests that already define the current `ModelTurnResult`, `ToolRequest`, approval, artifact, context-manifest, and append-only message invariants so Spec 009 preserves those seams instead of redefining them.
2. Add high-risk unit tests first for provider response translation in `src/providers/models.py`, covering plain-text replies, single allowed tool calls, multiple allowed tool calls up to the configured cap, backend-safe correlation ID generation when the provider omits one, unknown tool names, excessive tool-call fan-out, missing required arguments, and non-object tool arguments.
3. Add high-risk prompt-rendering tests first for `src/graphs/prompts.py`, proving the backend-owned structured prompt payload includes the required sections `system_instructions`, `conversation`, `attachments`, `tools`, `approval_guidance`, `response_contract`, and `metadata`, while preserving bounded transcript context, summary text already present in `AssistantState.messages`, normalized attachment metadata from the manifest, visible-tool guidance, and backend-authored approval rules.
4. Add high-risk state-contract tests first for `src/graphs/state.py` and `src/graphs/prompts.py`, proving the graph can carry one typed backend-owned prompt payload on `AssistantState` without changing the `complete_turn(state, available_tools)` adapter signature, and that `available_tools` remains the name-only execution surface rather than the full prompt-definition surface.
5. Add high-risk settings tests first for `src/config/settings.py`, covering default `runtime_mode=rule_based`, explicit provider-backed mode selection, missing required credentials, invalid timeout or retry values, invalid max-tool-request settings, supported `llm_tool_call_mode` values, `llm_disable_tools`, nullable `llm_base_url` and `llm_max_output_tokens`, and deliberate scaffold-mode fallback without silent adapter selection.
6. Add focused failure-classification tests for provider timeout, provider unavailable, auth failure, rate limiting, malformed provider payloads, and provider tool-schema errors, proving transport-class failures remain retryable run failures while malformed semantic output is treated as a safe completion path rather than an infrastructure retry.
7. Add provider-error boundary tests first, proving the provider module exposes one bounded error contract with category, retryable flag, and safe detail so worker retry logic does not rely on provider-SDK string parsing.
8. Add the provider SDK dependency in `pyproject.toml` and extend `.env.example` plus `src/config/settings.py` with the explicit provider-runtime settings required by the spec: `runtime_mode`, `llm_provider`, `llm_api_key`, `llm_base_url`, `llm_model`, `llm_timeout_seconds`, `llm_max_retries`, `llm_temperature`, `llm_max_output_tokens`, `llm_tool_call_mode`, `llm_max_tool_requests_per_turn`, and `llm_disable_tools`.
9. Update `apps/gateway/deps.py` to select the configured model adapter explicitly, keep `RuleBasedModelAdapter` available for scaffold mode, fail clearly during startup or graph construction when provider-backed mode is selected without required credentials, and avoid any silent fallback from provider-backed mode to rule-based behavior.
10. Refactor `src/providers/models.py` to preserve `ModelAdapter` and `RuleBasedModelAdapter` while adding one provider-backed adapter plus provider-local helpers for prompt serialization, timeout handling, bounded retry handling, response parsing, failure mapping, sanitization, translation into backend-owned `ModelTurnResult` and `ToolRequest` values, and one bounded provider-error contract without leaking provider-native objects outside the module boundary.
11. Replace the current prompt rendering in `src/graphs/prompts.py` with a structured, typed, backend-authored prompt builder that remains pure, deterministic, testable, and independent of direct database access, leaving provider-specific serialization inside the adapter layer.
12. Make the smallest additive changes needed in `src/graphs/state.py` so a typed prompt payload can flow into the provider adapter on `AssistantState`, and bounded provider execution metadata can flow back on `ModelTurnResult`, without breaking existing callers or introducing a second transcript or inference-history source of truth.
13. Update `src/graphs/nodes.py` to keep backend ownership of tool execution, assistant message persistence, tool artifacts, tool audit events, and context-manifest persistence while attaching the typed prompt payload before provider execution and passing the provider adapter only the bounded prompt inputs plus the visible-tool name list already authorized by backend policy.
14. Extend `src/graphs/nodes.py` and `src/policies/service.py` together so deterministic `approve` and `revoke` handling still wins before any provider interpretation, denied or unregistered tools are rejected fail-closed, and provider-suggested governed tools still require exact active approval before execution.
15. Reuse the existing governance lifecycle in `src/sessions/repository.py` and `src/graphs/nodes.py` so LLM-originated governed tool requests without exact approval create a governance proposal through the approval-owned persistence path, treat the resulting `proposal_id` as the canonical correlation identifier, persist the proposal packet and audit trail, and do not create any separate competing `tool_proposal` artifact first.
