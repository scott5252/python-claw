# Summary: Spec 007 Channels, Streaming, Chunking, and Media Pipeline

Spec 007 makes `python-claw` behave more like a real multi-channel assistant system without breaking the gateway-first architecture from the earlier specs. Before this slice, the system could accept inbound messages, run assistant logic, and create outbound intent artifacts, but it did not yet handle attachments in a structured way or send completed replies through a shared outbound delivery layer. After this slice, inbound messages can include canonical `attachments`, the worker can normalize those attachments into safe stored media records, and outbound replies can be dispatched through channel-aware logic for `webchat`, `slack`, and `telegram`.

One of the most important changes is how attachments are handled. The gateway now accepts attachment metadata as part of `POST /inbound/message`, but it does not download or process files during the HTTP request. Instead, it stores the accepted attachment references in durable database rows and still returns `202 Accepted` quickly. Later, when the worker claims the queued run, it performs first-pass normalization: it checks URL scheme, MIME type, and size limits, classifies the media type, stages the content into a runtime-owned storage path, computes a hash, and records the result in append-only attachment tables. This keeps the request path fast and durable while making media handling safer and easier to audit.

Spec 007 also adds a shared outbound dispatcher so reply delivery happens in one place instead of being hidden inside tools or channel adapters. The assistant runtime can still create runtime-owned outbound intents, but the worker now uses a dispatcher to finish the job after the assistant turn is complete. That dispatcher parses supported directives, strips them from user-visible text, applies channel capability rules, chunks long messages into deterministic blocks, creates durable outbound delivery rows, and then calls thin channel adapters to send one chunk or one media item at a time. This is important because it keeps transport logic out of the graph and makes retries, failures, and replay behavior much more predictable.

Another key improvement is delivery and media auditing. The system now records accepted attachment inputs, normalized attachment results, logical outbound deliveries, and per-attempt send records in dedicated append-only tables. That means developers and operators can inspect what happened during media normalization or outbound sending without relying only on logs. If a message is split into several chunks, each chunk gets its own durable delivery record. If a send fails and is retried, the retry is recorded as a new attempt on the same logical delivery instead of creating confusing duplicate rows.

Just as important, this spec keeps the scope intentionally bounded. In this phase, "streaming" means post-turn chunked delivery, not token-by-token live streaming. The adapters for `webchat`, `slack`, and `telegram` are thin integration points, not full provider clients with rich layouts or live presence features. The media pipeline normalizes and stores safe references, but it does not yet do OCR, transcription, or semantic media understanding. In practical terms, Spec 007 gives the project a strong foundation for realistic channels and media handling while still preserving the earlier rules around gateway ownership, worker execution, policy checks, and append-only durability.
